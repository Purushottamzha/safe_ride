# SafeRide Nepal v1.0 Product Scope

SafeRide Nepal v1.0 should be treated as a commercial school transport
management platform, not a demo attendance app. The existing architecture is
already strong enough to finish into a pilot-ready product if work stays
incremental and backward compatible.

## Product Positioning

Use this product description consistently:

**SafeRide Nepal - School Transport Management Platform**

Avoid positioning the project as only a QR attendance system. QR attendance is
one core workflow inside a larger transport operations platform.

## System Architecture

```text
Android Scanner / Door Scanner
        |
        v
MQTT Broker (Mosquitto)
        |
        v
NestJS Backend
        |
  +-----+----------+
  |     |          |
  v     v          v
Postgres Redis Socket.IO
  |                |
  +---------+------+------+
            |      |      |
            v      v      v
       Admin Web Parent Web Driver Web/App
```

Primary responsibilities:

- Scanner apps capture QR, GPS, and heartbeat events.
- MQTT transports device-originated events with offline retry support.
- NestJS validates, persists, authorizes, and broadcasts business events.
- PostgreSQL stores durable system records.
- Redis supports real-time scaling, cache, queues, and pub/sub patterns.
- Socket.IO updates admin, parent, and driver interfaces in real time.

## Real-Time Event Flow

```text
QR Scan
  |
  v
SQLite Offline Queue
  |
  v
MQTT Publish
  |
  v
NestJS MQTT Service
  |
  v
QR / Attendance Service
  |
  v
PostgreSQL
  |
  v
Notification + Tracking Gateways
  |
  v
Socket.IO
  |
  +--> Admin Dashboard / Control Center
  +--> Parent Dashboard
  +--> Driver Dashboard
```

GPS and heartbeat events follow the same intake path, but are handled by
tracking/device-monitoring services instead of attendance logic.

## Folder Architecture

```text

admin-web/        React admin portal, control center, CRUD, analytics, reports
parent-web/       React parent portal, child dashboard, tracking, notifications
driver-app/       Driver mobile app workspace

shared/           Shared contracts and reusable utilities
docs/             Product, deployment, device, and environment documentation
docker/           Mosquitto and infrastructure configuration
scripts/          Operational scripts for credentials, devices, and setup
```

Planned naming note: if a web-based driver portal is added, use `driver-web/`.
The current repository already has `driver-app/` for driver-facing work.


## v1.0 Modules

### 1. Authentication and Security Access

Goal: production-grade identity and role access.

Required:

- Admin, transport manager, school staff, parent, driver, and operator access
- JWT access tokens and refresh tokens
- RBAC and permissions enforced on protected backend routes
- Audit log for sensitive actions
- Session history or last-login visibility
- Password reset flow
- 2FA-ready data model and UI placeholders

### 2. Student Management

Goal: complete student lifecycle management.

Required:

- CRUD, archive, and soft delete
- Photo, class, section, roll/student ID, QR token, guardian links
- Emergency contacts and medical information
- Pickup stop, drop stop, assigned route, assigned bus
- Attendance history and scan timeline
- Bulk import, bulk export, bulk QR generation, QR print flow
- Search, filters, and operational detail page

### 3. Parent Portal

Goal: polished daily-use parent experience.

Required:

- Parent dashboard with child cards
- Attendance status card
- Assigned bus and driver card
- Live map and ETA
- Trip timeline and history
- Notifications and unread state
- Attendance calendar
- Profile, settings, and dark mode

### 4. Driver Portal

Goal: let drivers operate trips without admin help.

Required:

- Today's assigned route
- Stop sequence and student manifest
- Student attendance state
- Scanner health
- Trip start, trip complete, and breakdown report actions
- Emergency action

### 5. Fleet Management

Goal: manage buses as operational assets.

Required:

- Bus CRUD
- Capacity, registration, insurance, mileage, fuel type
- Maintenance status and schedule
- GPS and scanner assignment
- Current driver, current route, current occupancy, current status

### 6. Route Management

Goal: map-first route planning.

Required:

- Leaflet and OpenStreetMap
- OSRM route distance and ETA
- Route creation and editing
- Stops, pickup points, drop points
- Route replay
- Geofence support using open-source spatial tooling

### 7. Transport Control Center

Goal: flagship operations room for the transport office.

Required:

- Full-screen live map
- Every active bus visible
- Animated bus movement and heading rotation
- Status, driver, route, occupancy, scanner, GPS, battery, heartbeat
- Emergency visibility
- Route deviation and offline alerts
- Search and filter by bus, route, driver, status, emergency

### 8. Live Tracking

Goal: one GPS backend powering parent, admin, and driver views.

Required:

- Smooth marker interpolation
- Bus heading rotation
- Shared event model through MQTT and Socket.IO
- Parent-restricted tracking for only assigned children
- Admin-wide tracking for operations

### 9. Attendance

Goal: reliable school transport attendance.

Required:

- Morning and afternoon trips
- Board and drop scans
- Manual correction with audit trail
- Duplicate protection
- Offline recovery
- Export and statistics
- Student timeline

### 10. Notification Center

Goal: real-time operational awareness.

Required:

- MQTT event intake
- Socket.IO browser delivery
- Browser notifications where supported
- Push-ready backend design
- Events for boarded, reached school, delayed, emergency, scanner offline,
  GPS lost, driver changed, trip started, and trip completed

### 11. Analytics

Goal: useful transport and attendance reporting.

Required:

- Attendance percentage and late percentage
- Bus utilization
- Driver score
- Route efficiency
- Average pickup and drop time
- Monthly trends
- CSV, Excel, and PDF exports

### 12. Device Monitoring

Goal: make scanner health visible before failures affect students.

Required:

- Android scanner registry
- Battery, RAM, storage, heartbeat, internet, GPS accuracy, MQTT, camera status
- App/version metadata
- Logs
- Remote restart marked as future support until hardware-safe

### 13. System Settings

Goal: make the platform configurable per school.

Required:

- School profile
- Academic year
- Branding
- Language-ready labels
- Notification templates
- MQTT, scanner, GPS, and map settings
- Roles and permissions management
- Backup and restore process

### 14. Platform Security

Goal: safe enough for a real school pilot.

Required:

- Helmet
- Rate limiting
- Request validation
- Audit log
- Soft delete
- Password policy
- Encryption for sensitive secrets
- CSRF protection where applicable
- XSS protection in web apps
- Prisma-backed SQL injection protection
- RBAC on backend routes, not only the UI

## Digital Twin Principle

The platform should converge on three synchronized views:

- **Operations View:** transport office sees every bus and operational state.
- **Parent View:** parents see only their child's bus, ETA, and events.
- **Driver View:** drivers see their assigned route, stops, manifest, and trip controls.

One scan or GPS update should flow through MQTT/backend persistence/Socket.IO and
update all relevant views in real time.

## Implementation Sequence

1. Stabilize and document the existing backend contracts.
2. Finish the Transport Control Center as the flagship admin screen.
3. Upgrade admin student, fleet, route, attendance, and device workflows.
4. Upgrade the parent portal around live tracking and notifications.
5. Add or complete the driver portal.
6. Complete exports, analytics, and monitoring.
7. Harden security, backups, deployment, and integration testing.

## Definition of Done for v1.0

Version 1.0 is complete only if:

- Android scanner works.
- Offline sync works.
- MQTT works.
- GPS works.
- Attendance works.
- Admin CRUD works.
- Parent portal works.
- Driver portal works.
- Live tracking works.
- Notifications work.
- Reports work.
- Security passes.
- Docker deployment works.
- Complete documentation exists.

Every completed module must also meet these conditions:

- Existing scanner, MQTT, attendance, auth, Docker, and seed flows still work.
- Backend endpoints have validation and authorization.
- New database fields have Prisma migrations and indexes where needed.
- UI is responsive on laptop and mobile widths.
- Loading, empty, and error states are handled.
- Real-time features degrade gracefully when sockets or MQTT are offline.
- Audit logs capture sensitive create/update/delete/manual-correction actions.
- Exports do not expose data across schools.
- Build and type checks pass for affected apps.
- Changes are documented in this file or a module-specific doc.

## Pilot Readiness Checklist

Before a real school pilot:

- Replace default secrets in all environments.
- Review child-data retention and parental consent requirements.
- Confirm backup and restore procedure.
- Test scanner offline queue recovery with real devices.
- Test GPS loss, scanner offline, duplicate scans, and delayed network cases.
- Verify role permissions with admin, staff, operator, parent, and driver users.
- Run a full morning and afternoon simulated trip from scanner to parent portal.
